## Работа отклоняется от проверки

<div class="paragraph">

Пока эти проблемы не будут исправлены, ревьюер не примет работу на проверку и не даст обратную связь по всему проекту:

</div>

  - при запуске приложения в инструментах разработчика показываются ошибки,
  - внешний вид приложения не совпадает с [макетом,](Mesto.fig)
  - часть функциональности приложения не реализована,
  - на повторных итерациях не исправлены критические замечания ревьюера,
  - работа содержит вопросы или просьбы о помощи к ревьюеру.

## Критические требования

<div class="paragraph">

Без соблюдения этих требований ваш проект не пройдёт проверку, эти пункты обязательно попросит исправить ревьюер. Давайте повторим, как всё должно выглядеть.

</div>

### **Стиль написания кода**

   
    

    
    Форматирование кода:
    

    
  - код проекта отформатирован единообразно,
  - нет нарушений иерархии отступов.

  - 
    

    
    Объявление переменных:
    

    
  - для объявления переменных используется `let` и `const`, не используется объявление через `var`;
  - если переменная объявлена через `let`, её значение должно быть напрямую перезаписано;
  - если переменная не перезаписывает своё значение, она объявлена через `const`.

  - 
    

    
    Требования по именованию переменных:
    

    
  - имена переменных и функций написаны в стиле `camelCase`;
  - имена переменных — существительные, отражающие то, что в них хранится; если в проекте есть несколько переменных с похожими данными, по их наименованиям понятно, что хранится в каждой переменной;
  - имена массивов — существительные во множественном числе;
  - имена коллекций NodeList — существительные во множественном числе;
  - имена функций начинаются с глагола, отражающего то, что они делают;
  - при именовании не используются транслит и неуместные сокращения.

  - 
    

    
    JavaScript-код проекта разбит по отдельным файлам в зависимости от их функциональности:
    

    
  - `src/scripts/components/card.js` — функции для работы с карточками;
  - `src/scripts/components/modal.js` — функции для работы модальных окон;
  - `src/scripts/components/validation.js` — функции для работы валидации; из файла экспортируется только функция активации валидации `enableValidation` и функция очистки ошибок валидации `clearValidation`;
  - `src/scripts/components/api.js` — функции запросов к API-серверу;
  - `src/scripts/index.js` — точка входа в приложение.

### **Соответствие заданию**

  - Проект выполнен на основе существующего проекта **Mesto**.

  - Проект инициализирован, и в нём есть файл `package.json`.

  - К проекту подключён сборщик Vite и библиотека для публикации проекта на GitHub Pages. Эти зависимости находятся в файле `package.json` в блоке `devDependencies`.

  - 
    

    
    Файл `package.json` содержит команды для запуска локального сервера разработки, сборки и публикации проекта. При выполнении в терминале команды:
    

    
  - `npm run dev` — запускается локальный сервер разработки и страница проекта автоматически открывается в браузере;
  - `npm run build` — проект собирается и в корне проекта появляется директория `dist`;
  - `npm run deploy` — собранный проект публикуется на GitHub Pages.
    

    
    При запуске локального сервера разработки, сборке и публикации проекта в терминале не должно быть ошибок.
    


  - 
    

    
    В корне проекта есть файл конфигурации сборщика `vite.config.js` с правилами:
    

    
  - `open: true`  — для автоматического открытия страницы проекта в браузере при запуске dev-сервера;
  - `base: './'` — для определения относительного пути подключения ресурсов.

  - При открытии страницы с сервера запрашиваются данные карточек и пользователя и отображаются на странице.

  - 
    

    
    Корректно работает обновление аватара пользователя:
    

    
  - при наведении указателя мыши на аватар на нём появляется иконка редактирования, а при клике открывается модальное окно с формой обновления аватара;
  - при открытии модального окна обновления аватара поле формы должно быть пустым и не должно присутствовать ошибки валидации, которая могла остаться с прошлого открытия;
  - кнопка формы должна оставаться неактивной до ввода данных;
  - при нажатии кнопки «Сохранить» на сервер отправляется запрос на обновление аватара, а текст кнопки меняется на «Сохранение…», после выполнения запроса текст кнопки обратно меняется на «Сохранить»; если запрос выполнен успешно, модальное окно закрывается и аватар обновляется на странице;
  - модальное окно можно закрыть, кликнув по кнопке закрытия или оверлею или нажав клавишу `Escape`, — в этом случае запрос не отправляется и аватар на странице не обновляется.

  - 
    

    
    Корректно работает редактирование профиля пользователя:
    

    
  - при нажатии на кнопку редактирования открывается модальное окно редактирования профиля;
  - при открытии модального окна редактирования профиля в поля формы автоматически подставляются текущие данные пользователя со страницы и не должно присутствовать ошибок валидации, которые могли остаться с прошлого открытия;
  - кнопка формы должна оставаться неактивной до редактирования данных пользователя;
  - при нажатии кнопки «Сохранить» на сервер отправляется запрос на обновление данных пользователя, а текст кнопки меняется на «Сохранение…», после выполнения запроса текст кнопки обратно меняется на «Сохранить»; если запрос выполнен успешно, модальное окно закрывается и данные пользователя обновляются на странице;
  - модальное окно можно закрыть, кликнув по кнопке закрытия или оверлею или нажав клавишу `Escape`, — в этом случае запрос не отправляется и данные пользователя на странице не обновляются.

  - 
    

    
    Корректно работает добавление новых карточек:
    

    
  - при нажатии на кнопку «+» открывается модальное окно добавления карточки;
  - при открытии модального окна добавления карточки поля формы должны быть пустыми и не должно присутствовать ошибок валидации, которые могли остаться с прошлого открытия;
  - кнопка формы должна оставаться неактивной до ввода данных;
  - при нажатии на кнопку «Создать» отправляется запрос на добавление карточки на сервер, а текст кнопки меняется на «Создание…», после выполнения запроса текст кнопки обратно меняется на «Создать»; если запрос выполнен успешно, модальное окно закрывается и в начало списка карточек добавляется новая с введёнными в форму данными;
  - модальное окно можно закрыть, кликнув по кнопке закрытия или оверлею или нажав клавишу `Escape`, — в этом случае запрос не отправляется и новая карточка не добавляется на страницу.

  - 
    

    
    Корректно работает функциональность карточек:
    

    
  - кнопка лайка имеет правильное состояние, зависящее от того, лайкнул ли её пользователь;
  - у карточки отображается количество лайков;
  - при нажатии на кнопку лайка карточки отправляется запрос на сервер и по его завершению кнопка лайка изменяет состояние и обновляется счетчик количества лайков;
  - при нажатии на изображение карточки открывается модальное окно с полноразмерным изображением и его описанием; также модальное окно можно закрыть, кликнув по кнопке закрытия или оверлею или нажав клавишу `Escape`.
  - кнопка удаления карточки отображается только у её автора;
  - при нажатии на кнопку удаления карточки отправляется запрос на сервер, и при его успешном завершении карточка удаляется со страницы;
  - если реализовано модальное окно подтверждения удаления карточки, то эта функциональность работает правильно: при нажатии на кнопку «Да» на сервер отправляется запрос на удаление карточки, а текст кнопки меняется на «Удаление…», после выполнения запроса текст кнопки обратно меняется на «Да»; если запрос выполнен успешно, модальное окно закрывается и соответствующая карточка удаляется со страницы; модальное окно также можно закрыть, кликнув по кнопке закрытия или оверлею или нажав клавишу `Escape`, — в этом случае запрос не отправляется и карточка не удаляется со страницы;

  - 
    

    
    Корректно работает валидация форм:
    

    
  - у форм добавления новой карточки, обновления аватара и редактирования данных пользователя работает валидация; если хотя бы одно из полей формы не прошло валидацию, то кнопка формы должна быть неактивной и под полем ввода, не прошедшим валидацию, должно показываться сообщение об ошибке; если ошибка не попадает под правила валидации, показывается стандартный браузерный текст ошибки;
    
  - в разметке форм отключена встроенная браузерная валидация, а кнопкам добавлен класс `popup__button_disabled` и атрибут `disabled`;
    
  - 
        
    
        
        в разметку полей ввода добавлены атрибуты, отвечающие за валидацию, после каждого поля ввода добавлен элемент для вывода текста ошибки. Правила валидации полей ввода:
        
    
        
  - все поля ввода обязательные;
  - поле `popup__input_type_avatar` должно содержать ссылку;
  - поле `popup__input_type_name` должно содержать от 2 до 40 символов и может включать только латинские и кириллические буквы, знаки дефиса и пробелы, для проверки символов должно использоваться регулярное выражение;
  - поле `popup__input_type_description` должно содержать от 2 до 200 символов;
  - поле `popup__input_type_card-name` должно содержать от 2 до 30 символов и может включать только латинские и кириллические буквы, знаки дефиса и пробелы;
  - поле`popup__input_type_url` должно содержать ссылку.
    
  - 
        
    
        
        код, отвечающий за валидацию, написан в модуле `src/scripts/components/validation.js`,
        
  - он должен быть универсальным
  - никаких конкретных селекторов для кнопок и полей ввода быть не должно; все селекторы, необходимые для валидации, нужно получать из объекта валидации, который передается в вызов каждой функции;
        
    
    
  - 
        
    
        
        модуль валидации содержит функции:
        
    
        
  - `showInputError` — отображает сообщение об ошибке под невалидным полем и добавляет соответствующие классы;
  - `hideInputError` — скрывает сообщение об ошибке и удаляет классы, связанные с ошибкой;
  - `checkInputValidity` — проверяет валидность конкретного поля; если оно невалидно, вызывает `showInputError`, иначе — `hideInputError`; в случае, если в поля «Имя» или «Название» введён любой символ, кроме латинской буквы, кириллической буквы и дефиса, выводит кастомное сообщение об ошибке;
  - `hasInvalidInput` — возвращает значение `true`, если хотя бы одно поле формы не прошло валидацию;
  - `disableSubmitButton` — делает кнопку формы неактивной;
  - `enableSubmitButton` — включает кнопку формы;
  - `toggleButtonState` — включает или отключает кнопку формы в зависимости от валидности всех полей;
  - `setEventListeners` — добавляет обработчики события `input` для всех полей формы, при каждом вводе проверяет валидность поля и вызывает функцию `toggleButtonState`;
  - `clearValidation` — очищает ошибки валидации формы и делает кнопку неактивной, принимает DOM-элемент формы и объект с настройками;
  - `enableValidation` — отвечает за включение валидации всех форм, принимает объект настроек, которые используются при валидации.

  - 
    

    
    Корректно работает взаимодействие с сервером:
    

    
  - в файле `src/scripts/components/api.js` написаны все функции запросов, описанные в задании: получение данных пользователя и карточек, обновление аватара и данных пользователя, запросы добавления, удаления и лайк карточек;
  - функции запросов к серверу возвращают Promise, в них нет кода, отвечающего за работу с DOM и изменение отображения страницы;
  - ответ сервера всегда проверяется на корректность проверкой res.ok;
  - действия с DOM-элементами на странице производятся только после завершения запроса;
  - в конце цепочки обработки каждого обращения к серверу есть обработка ошибок;
  - базовый адрес сервера, заголовок авторизации и типа данных запроса вынесены в отдельную переменную и переиспользуются.

  - 
    

    
    Другие требования:
    

    
  - из файла `index.html` удалены предустановленные значения ссылки на изображение аватара, имя и занятие пользователя;
  - проект опубликован на GitHub Pages и корректно работает;
  - в корне проекта есть оформленный файл `README.md`, содержащий ссылку на опубликованный проект и команды для запуска локального сервера, сборки и публикации проекта.

## Хорошие практики

<div class="paragraph">

В этом списке собраны приёмы, которые помогут сделать проект ещё лучше. Опытные разработчики обратят на них внимание.

</div>

  - Контентные изображения содержат `alt` с корректным описанием, которое соответствует языку страницы.

  - 
    

    
    Функции выполняют только одну задачу:
    

    
  - функция создания карточки возвращает DOM-элемент карточки, но не вставляет его в разметку;
  - функции модальных окон не содержат код для заполнения или очистки формы;
  - функции валидации не содержат код не относящийся к валидации;

  - Для перебора элементов массива используется метод `.forEach` или цикл `for…of`.

  - Все DOM-элементы, необходимые для реализации логики, вынесены в переменные.

  - Для поиска DOM-элементов используется метод `querySelector`.

  - Для вставки данных на страницу не используется свойство `innerHTML`.

  - 
    

    
    Нет дублирования кода, например:
    

    
  - поиск одного и того же DOM-элемента происходит только один раз;
  - повторяющаяся логика вынесена в отдельные функции, которые переиспользуются в соответствующих местах.

  - Операции над DOM-элементами выполняются до их вставки в разметку.

  - Код логично организован, например, сначала идёт объявление переменных, затем объявление функций и в конце вызывается отображение данных на странице.

  - Нет лишнего кода: переменных, которые не используются, закомментированных фрагментов, избыточной логики. А также удалены все `console.log`, `alert` и `debugger`, которые используются во время отладки.

  - 
    

    
    Нет утечек памяти, например:
    

    
  - обработчики навешиваются только один раз при инициализации приложения;
  - валидация активируется только один раз, а не при каждом открытии модального окна.
